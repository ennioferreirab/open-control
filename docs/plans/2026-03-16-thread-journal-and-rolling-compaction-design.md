# Thread Journal And Rolling Compaction Design

## Summary

Add a task-scoped Markdown journal that mirrors the complete task thread and workflow history without replacing Convex as the source of truth. Agents keep using the Convex thread as their primary operational context, but they also receive a stable absolute path to a full-fidelity `THREAD_JOURNAL.md` file plus a rolling compacted summary that can grow indefinitely while staying under a configurable size ceiling.

## Decisions

| Decision | Choice |
|----------|--------|
| Canonical thread source | Convex `messages` remains primary |
| Full-history artifact | Task-local `THREAD_JOURNAL.md` in the task output directory |
| Journal scope | One journal per task, including workflow, thread, and step events |
| Journal path | `~/.nanobot/tasks/<safe_task_id>/output/THREAD_JOURNAL.md` |
| Compaction scope | Derived context only; raw Convex messages remain untouched |
| Compaction model | LLM-only rolling summary, no deterministic fallback |
| Failure policy | Log error and transition task to `crashed` |
| Trigger policy | Background compaction when message-count or char-count threshold is exceeded |
| Persisted derived state | Local JSON file adjacent to the journal, not a Convex schema field |

## Goals

- Keep the complete task/workflow/step conversation available in a human-readable file.
- Make agents aware of that file through prompt metadata and explicit instructions.
- Prevent thread context from growing without bound by maintaining a rolling compacted summary.
- Preserve important historical facts, decisions, files, and workflow transitions across repeated compactions.
- Reuse existing background-task patterns so compaction behaves like memory consolidation.

## Non-Goals

- Replacing Convex as the source of truth for thread history.
- Mutating or deleting existing Convex messages during compaction.
- Introducing a fallback summarizer when the LLM compaction fails.
- Building a UI for browsing the journal in the first implementation wave.

## Architecture

### Storage Layout

Store task-scoped derived artifacts next to existing task output files:

- `~/.nanobot/tasks/<safe_task_id>/output/THREAD_JOURNAL.md`
- `~/.nanobot/tasks/<safe_task_id>/output/THREAD_COMPACTION_STATE.json`

The Markdown file is the full readable journal. The JSON state stores the rolling compaction metadata needed to continue compacting without touching Convex or reparsing the entire journal every turn.

### Journal Ownership

Convex remains the canonical event log. A new backend service materializes a local mirror from those messages:

- `ThreadJournalService` orchestrates journal sync and compaction.
- A small file-store layer owns atomic reads/writes for the Markdown journal and JSON compaction state.
- Execution and conversation context builders call this service before assembling prompt context.

This keeps the journal local, derived, and recoverable from Convex if needed.

### Journal Structure

`THREAD_JOURNAL.md` should have two types of content:

1. Rewritten stateful sections:
   - `Task Snapshot`
   - `Execution Plan`
2. Append-only event log:
   - user messages
   - lead-agent plan/chat messages
   - step completions
   - system errors
   - thread compaction events

Recommended shape:

```md
# Thread Journal
- Task: ...
- Task ID: ...
- Created: ...
- Last Updated: ...

## Task Snapshot
- Status: ...
- Assigned Agent: ...
- Board: ...
- Workflow Spec: ...
- Plan Generated By: ...

## Execution Plan
... current plan rendered as markdown ...

## Thread Events

### 2026-03-16T12:00:00Z | user | user_message
...

### 2026-03-16T12:10:55Z | dev-agent | step_completion | step=<step_id>
...

### 2026-03-16T12:15:00Z | system | thread_compaction
Compacted messages: 5
Key points:
- ...
```

## Sync Model

The journal must stay complete even though some messages originate outside Python. The sync model therefore has two entry points:

1. Best-effort sync on backend-owned thread writes:
   - `send_message`
   - `post_step_completion`
   - `post_lead_agent_message`
   - system error posting paths
2. Reconciliation sync on thread reads:
   - execution context building
   - conversation context building
   - mention/follow-up handlers

The reconciliation path fetches the full Convex thread, appends any unseen messages to the journal, and updates the local compaction state. This guarantees eventual consistency even when a user message is created directly through the dashboard.

## Rolling Compaction Model

### Live Context Shape

Agents receive three thread-related sections:

1. `[Compacted Thread Summary]`
2. `[Recent Thread Window]`
3. `[Thread Journal]`

The recent window keeps the last `15` raw messages. Older eligible messages are absorbed into a rolling `compacted_summary`.

### Persisted State

`THREAD_COMPACTION_STATE.json` should track:

- `last_compacted_message_id`
- `last_compacted_timestamp`
- `compacted_summary`
- `source_message_ids`
- `message_count_since_compaction`
- `char_count_since_compaction`
- `summary_token_budget`
- `recent_window_messages`

This state is derived and can be rebuilt from Convex + journal if damaged.

### Recompression Rule

Each compaction cycle consumes:

- the previous `compacted_summary`
- the next eligible batch of older messages
- important structured events that cannot be silently dropped

It produces a new `compacted_summary`. This makes compaction recursive and effectively unbounded in time while the summary itself stays below a configured maximum token budget.

### Structured Events That Must Survive Semantically

Even when old, these must remain represented in the rolling summary:

- `lead_agent_plan`
- `lead_agent_chat` when it carries material decisions
- `step_completion`
- `system_error`
- approval/denial/review decisions
- messages with file attachments or artifacts

If any of these fall into a batch being compacted, the LLM prompt must instruct the model to preserve their facts explicitly.

## Thresholds And Configuration

Recommended environment variables:

- `MC_THREAD_JOURNAL_ENABLED=true`
- `MC_THREAD_RECENT_WINDOW_MESSAGES=15`
- `MC_THREAD_COMPACTION_BATCH_MESSAGES=5`
- `MC_THREAD_COMPACTION_TRIGGER_MESSAGES`
- `MC_THREAD_COMPACTION_TRIGGER_CHARS`
- `MC_THREAD_COMPACTED_SUMMARY_MAX_TOKENS=20000`

The trigger thresholds decide when to schedule compaction. The summary max token value is a hard upper bound, not a target size.

## Failure Handling

Compaction is mandatory, not best-effort.

If the LLM compaction fails:

1. log the exception with task id and journal path
2. optionally post a system message describing the compaction failure
3. transition the task to `crashed`
4. leave the journal and compaction state untouched for inspection

There is no deterministic fallback. The runtime must not continue with degraded or potentially misleading compressed context.

## Integration Points

### Context Assembly

Update the execution request pipeline so task and step prompts include:

- the absolute `thread_journal_path`
- the current `compacted_summary`
- the recent 15-message raw window
- explicit instructions that the complete thread history is available in the journal

This should flow through the shared thread-context builder so execution, mentions, and conversation services stay aligned.

### Background Scheduling

Reuse `mc.application.execution.background_tasks.create_background_task` for thread compaction scheduling. Deduplicate by task id so repeated threshold crossings do not start parallel compactions for the same task.

### Task Crashes

The compaction service needs the bridge so it can call the existing task status mutation path and move the task to `crashed` when compaction fails.

## Testing Strategy

Required coverage:

- journal path resolution under the task output directory
- journal creation and append behavior for user, plan, step, and system events
- reconciliation sync appending unseen Convex messages
- rolling compaction reusing the previous `compacted_summary`
- summary budget enforcement through repeated recompression
- recent-window preservation at 15 messages
- journal hint injection into task, step, and conversation contexts
- compaction-task deduplication
- compaction failure crashing the task
- raw Convex message history remaining unchanged
