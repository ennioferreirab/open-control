# Convex Code Conventions

Applies to `dashboard/convex/`. The Convex backend runs locally (`npx convex dev --local`) on port 3210.

> **Cross-service naming**: see [`cross_service_naming.md`](cross_service_naming.md) for the shared naming contract between Python, Convex, and TypeScript.

## Architecture

```
dashboard/convex/
├── *.ts              # Thin Convex function registrations (query, mutation, etc.)
├── lib/              # Pure business logic — no Convex function builders
├── _generated/       # Auto-generated by Convex (never edit)
└── schema.ts         # Single source of truth for all table definitions
```

**Rule**: top-level `convex/*.ts` files are thin handlers. All business logic lives in `convex/lib/`. Handlers validate input, delegate to `lib/`, and return results. No complex logic in handlers.

```typescript
// CORRECT — thin handler delegates to lib
export const create = mutation({
  args: { title: v.string(), boardId: v.id("boards") },
  handler: async (ctx, args) => {
    return await createTask(ctx, args);
  },
});

// WRONG — business logic in handler
export const create = mutation({
  args: { title: v.string(), boardId: v.id("boards") },
  handler: async (ctx, args) => {
    const board = await ctx.db.get(args.boardId);
    if (!board) throw new ConvexError("Board not found");
    const agents = await ctx.db.query("agents")...
    // ... 50 lines of logic ...
  },
});
```

## Boundary

`convex/` is a self-contained backend module:

- **No imports from outside `convex/`** — do not import from `../lib/`, `../components/`, `../features/`, etc. If you need a pure function, move it into `convex/lib/`.
- **Only import from**: `convex/values`, `convex/server`, `./_generated/*`, `./lib/*`, `./schema`.
- **Architecture test should enforce this** — verify that `convex/` files only import from allowed sources.

## Naming

| Element | Convention | Example |
|---------|-----------|---------|
| Files | camelCase | `boardSquadBindings.ts` |
| Convex functions | camelCase | `getById`, `listByStatus`, `softDelete` |
| Table names | camelCase | `tasks`, `boardSquadBindings`, `runtimeClaims` |
| Index names | `by_` prefix | `by_taskId`, `by_status`, `by_boardId_squadSpecId` |
| Lib modules | camelCase | `taskLifecycle.ts`, `stepTransitions.ts` |
| TypeScript types | PascalCase | `TaskSnapshot`, `TransitionResult` |
| Validators (exported) | camelCase + `Validator` | `taskStatusValidator` |

## Schema

- **Single `schema.ts` file** — all tables defined in one place with `defineSchema` / `defineTable`.
- **Extract shared validators** and export them from `schema.ts`:
  ```typescript
  export const taskStatusValidator = v.union(
    v.literal("pending"),
    v.literal("running"),
    v.literal("completed"),
    v.literal("failed"),
  );
  ```
  Reference these in mutation args instead of redefining inline. This prevents drift between schema and function arguments.
- **Timestamps**: ISO 8601 strings (`v.string()`), not numeric timestamps.
- **Soft-delete**: `deletedAt: v.optional(v.string())` pattern. Queries must filter soft-deleted records.
- **No `v.any()`** — use typed validators. Acceptable only for truly polymorphic payloads with a documented reason. Prefer `v.object()`, `v.record()`, or discriminated unions.

## Argument Validation

- **Use `v.id("tableName")`** for document ID arguments — never `v.string()` with a cast.
- **Use shared validators** for status fields — never inline union literals.
- **Validate at the boundary**: every public `query`/`mutation` must validate all arguments with `v.*` validators.
- **Optional arguments**: use `v.optional()` — do not use `undefined` checks.

```typescript
// CORRECT
args: { taskId: v.id("tasks"), status: taskStatusValidator }

// WRONG — string with cast
args: { taskId: v.string() }  // then: args.taskId as Id<"tasks">
```

## Error Handling

- **`ConvexError`** (from `convex/values`) for all user-facing validation errors:
  ```typescript
  import { ConvexError } from "convex/values";
  throw new ConvexError("Task not found");
  throw new ConvexError({ code: "INVALID_TRANSITION", from: "pending", to: "completed" });
  ```
- **`Error`** only for programmer errors (bugs, unreachable code, invariant violations).
- **Rule of thumb**: if a client can trigger it, use `ConvexError`. If it means the code is broken, use `Error`.
- **Structured errors**: for errors that the frontend needs to handle programmatically, pass an object to `ConvexError` instead of a plain string.

## Function Visibility

| Decorator | Use |
|-----------|-----|
| `query` / `mutation` | Public API — callable from frontend |
| `internalQuery` / `internalMutation` | Python gateway only — not exposed to frontend |
| `action` / `internalAction` | Side-effects, external API calls (use sparingly) |

**Guidelines:**
- Default to `internal*` for new functions. Only make public if the frontend needs it.
- Queries must be **side-effect free** — no writes, no external calls.
- Mutations must be **deterministic** — no `Math.random()`, no `Date.now()` (use `Date.now()` from Convex scheduler context).
- Use `action` only when you need to call external APIs. Prefer mutation + scheduled action for complex flows.

## Type Safety

- **`Id<"tableName">`** for all document references — never plain `string`.
- **Type-only imports**: use `import type { ... }` for types from `_generated/`.
- **Custom context types** in `lib/` modules to narrow the full Convex context:
  ```typescript
  type TransitionMutationCtx = {
    db: { get: ...; patch: ...; };
  };
  ```
  This makes `lib/` modules testable without the full Convex runtime.
- **Minimize `as` casts** in production code. If you need a cast, consider:
  - Adding a proper type guard function
  - Defining an intermediate type
  - Using a conversion function
  - Only cast in one place and propagate the typed result
- **Null checks**: always handle `null` from `ctx.db.get()`. Never assume a document exists.

```typescript
// CORRECT — explicit null check
const task = await ctx.db.get(args.taskId);
if (task === null) {
  throw new ConvexError("Task not found");
}
// task is now narrowed to non-null

// WRONG — unchecked access
const task = await ctx.db.get(args.taskId);
return task.title; // runtime error if null
```

## Query Patterns

### Index usage

Always use indexes for queries. Never use `ctx.db.query("table").filter()` without an index:

```typescript
// CORRECT — uses index
const tasks = await ctx.db
  .query("tasks")
  .withIndex("by_status", (q) => q.eq("status", "running"))
  .collect();

// WRONG — full table scan
const tasks = await ctx.db
  .query("tasks")
  .filter((q) => q.eq(q.field("status"), "running"))
  .collect();
```

### Pagination

For large result sets, use `.paginate()` instead of `.collect()`:

```typescript
export const list = query({
  args: { paginationOpts: paginationOptsValidator },
  handler: async (ctx, args) => {
    return await ctx.db
      .query("tasks")
      .withIndex("by_createdAt")
      .order("desc")
      .paginate(args.paginationOpts);
  },
});
```

### Read-after-write consistency

Convex mutations are transactional. Reads within a mutation always see writes from the same mutation. No need for explicit refresh or cache invalidation.

## Idempotency

Use the `runtimeReceipts` pattern for mutations that may be retried:

1. Accept optional `idempotencyKey` argument
2. Check for existing receipt before executing
3. Store receipt after completion
4. Return existing result if receipt found

## State Machines

Use the optimistic-concurrency transition pattern:

1. Accept `fromStatus` + `stateVersion` in args
2. Verify current state matches expected state
3. Return discriminated union: `"applied"` | `"noop"` | `"conflict"`
4. Never apply a transition without checking the guard

```typescript
type TransitionResult =
  | { outcome: "applied"; newVersion: number }
  | { outcome: "noop" }
  | { outcome: "conflict"; currentStatus: string; currentVersion: number };
```

## Testing

- **Test runner**: `npm run test` (vitest)
- **Co-located tests**: `foo.test.ts` next to `foo.ts`
- **Use typed test helpers**: `as unknown as Id<"tasks">` for branded ID mocks.
- **Test `lib/` modules directly** — they are pure functions with narrowed context types, making them easy to unit test.
- **Architecture test**: verify that `convex/` files only import from allowed sources.
